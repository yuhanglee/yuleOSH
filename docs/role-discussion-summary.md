# yuleOSH v0.1.0 三方角色讨论报告

> 日期: 2026-06-05
> 讨论形式: 系统工程师(A) + 架构专家(B) + 开发专家(C) 各自分析后交叉点评
> 发起人: 明天华

---

## 一、各角色核心观点摘要

### 🧑‍🔧 系统工程师 A — 「客户需求第一」

**核心判断：yuleOSH 的当前需求优先级偏了——太关注工具链本身，忽略了嵌入式团队的真正痛点。**

| 议题 | 观点 |
|------|------|
| ASPICE | 真正的门槛不是文档和工具，是**流程纪律和团队执行力**。yuleOSH 提供了"形"但缺"神" |
| 质量关键 | 嵌入式软件质量最大的敌人不是代码 bug，是**需求模糊和沟通断层**。OpenSpec 的方向是对的，但团队需要养成用它的习惯 |
| 测试缺口 | 最缺的是 **集成测试环境**——MCU 的 IO 交互、中断时序、外设驱动，这些东西在纯软件环境里测不出来 |
| 台架接入 | 嵌入式台架连接本质是 **物理世界问题**，不是软件问题。需要定义一套"Test Adapter Interface"，统一不同台架的通信协议 |
| 客户真需求 | 嵌入式团队最痛的是：**需求改一次→测试重做一次→台架重新跑一次→文档重新写一次**。yuleOSH 的诞生思路是对的，但要帮团队把"从需求变更到重新验证"的闭环跑通 |

**给其他角色的挑战：**
- 给架构师：你设计的再漂亮的架构，如果客户连 spec 都不想写，有什么意义？
- 给开发专家：你提升覆盖率到 70%，但如果覆盖的是不重要的代码，客户感受不到价值。

---

### 🏗️ 架构专家 B — 「架构先行」

**核心判断：yuleOSH 的架构能在 MVP 阶段撑住，但要商业化必须重构。API 层太胖、没有插件系统、没有事件总线是三大致命伤。**

| 议题 | 观点 |
|------|------|
| 架构评价 | 当前模块划分大体合理，但 `api/` 16 个模块依赖 `store/`、`ci/`、`pipeline/` 等所有模块——**API 层成了上帝总线** |
| HIL/SIL 方案 | 必须是 **Plugin 架构 + Adapter 模式**——定义 `TestBenchAdapter` 接口，每个台架（NI PXI / dSPACE / 自研）实现自己的 adapter，通过 yuleOSH API 注册上来 |
| 多租户 SaaS | 当前的 SQLite 单机架构不支持多租户。需要迁移到 PostgreSQL + 租户隔离策略（schema-per-tenant 或 row-level security） |
| 解耦建议 | `api/` 应拆成 gateway 层和业务层；`pipeline/` 和 `ci/` 有重叠，应该合并或明确定义边界 |
| 三层 CI/CD | **应该把 CI/CD 做成 Pipeline 的 plugin，而不是反过来。** 当前 ci/ 是独立模块，pipeline/ 是独立模块，中间没有标准接口 |

**给其他角色的挑战：**
- 给系统工程师：你说客户痛点，但如果底层架构不支持扩展，满足一个客户的需求就会让系统无法维护。你要的是一个月交付还是三年可维护？
- 给开发专家：你说 API 层零测试，我同意。但真正的问题是 API 层承担了太多职责，让它变得难测。解耦之后自然就能测了。

---

### 👨‍💻 开发专家 C — 「落地才是硬道理」

**核心判断：代码质量有亮点，但有几个致命问题——Pipeline 大泥球、AI Agent 名实不符、UI/API 层裸奔。**

| 议题 | 观点 |
|------|------|
| 代码质量 | 核心引擎很好（spec/validate 74.6% 覆盖），但 pipeline/run.py 625 行一个文件违反单一职责 |
| 测试缺口 | **真正的危机不是 39.8%——是 689 行测试 vs 7262 行源码（9.5%）。** UI/API 层 3385 行全部零测试 |
| 覆盖率 | 两个原因各占一半：测试没写够 + 代码结构难测（强耦合文件系统/subprocess，无依赖注入）|
| 台架落地 | **不要自研 HIL 框架**——那是无底洞。用 pytest + pyserial 做 SIL，200 行代码出闭环 |
| 名实不符 | **最尖锐的问题：标注了"AI Agent 流水线"，实际只是模板引擎。** 每个 pipeline run 生成漂亮的 Markdown，但没有一个"真的分析" |

**给其他角色的挑战：**
- 给系统工程师：你说"客户不需要改代码"，但一个工具的好坏最终体现在开发者用它的感受上。当前的上手流程需要理解 3 个新概念（OpenSpec + Superpowers 14 Rules + Harness Engineering），认知负荷过高。
- 给架构师：你说要 Plugin 架构，但在 v0.2.0 做 Plugin 系统是过度工程。先让核心功能跑通、有测试保护，再考虑扩展性。

## 二、三方交锋点整理

### 🔥 交锋 1：架构重构 vs 功能交付

| 立场 | 观点 |
|:-----|:------|
| B 架构师 | API 层必须重构，太胖了。需要 Plugin 系统来支持台架适配 |
| C 开发专家 | v0.2.0 做 Plugin 系统是过度工程。优先把名实不符的问题解决 |
| A 系统工程师 | 客户不管架构，只管能否解决问题 |

**结论：** 折中——不做大规模重构，但做最小化解耦（API 层引入 Router 接口、IO 层引入依赖注入）。目标是让代码可测，而不是完美。

### 🔥 交锋 2：覆盖率目标

| 立场 | 观点 |
|:-----|:------|
| C 开发专家 | 当前 39.8% 可接受。API 层要补测试（2 天到 50%+），但更重要的是问"测试覆盖了有价值的东西吗" |
| A 系统工程师 | 同意。覆盖率数字是手段不是目的。先覆盖核心流程 |
| B 架构师 | 同意。解耦后自然可测。先解耦再追覆盖率 |

**结论：** 三方一致——短期目标 50%+（补 API 层），中期目标 70%+（解耦后补齐），不盲目追数字。

### 🔥 交锋 3：台架接入方案

| 立场 | 观点 |
|:-----|:------|
| B 架构师 | Plugin 架构 + Adapter 模式，定义 TestBenchAdapter 接口 |
| C 开发专家 | Phase 1: pytest + pyserial 200 行出 SIL 闭环。不要一开始就搞厚重框架 |
| A 系统工程师 | 台架接入的核心难点不是技术，是**不同台架厂商的协议不统一**。需要定义通用协议 |

**结论：** 分阶段——Phase 1 轻量级 SIL（pytest），Phase 2 定义通用 Adapter 接口，Phase 3 接入真实 HIL 台架。

### 🔥 交锋 4：AI Agent 名实不符

| 立场 | 观点 |
|:-----|:------|
| C 开发专家 | **最严重的问题。** 当前 pipeline 只是模板引擎，标注了"Claude 开发""Hermes 审查"，实际没调用任何 AI |
| A 系统工程师 | 同意。Demo 好看但实际没用。要么真的集成 AI，要么改名 |
| B 架构师 | 同意。这是信任问题——用户用了发现是假的，就不会再信任了 |

**结论：** **三方一致最严重的的问题。** AI Agent 流水线要么真集成 LLM API，要么诚实地改名为 Template Pipeline。

## 四、最终建议（三方一致）

1. **短期（v0.2.0）：先把信任修好。** AI Agent 流水线说实话、API 层加测试保护
2. **中期（v0.3.0）：跑通台架。** SIL 先行，HIL 定接口
3. **长期（v1.0.0）：ASPICE 认证级。** 70%+ 覆盖率、Layer 2/3 全通、多租户、Plugin 生态

**当前最紧急的一件事：** AI Pipeline 名实不符 —— 要么集成 LLM，要么诚实改名。
