Architecture Overview

yuleOSH is built on a three-layer architecture that combines structured requirements management, AI agent-driven development, and automated CI/CD for embedded systems.

Layer 1 OpenSpec — Requirements Framework

RFC 2119-based structured requirements using SHALL / SHOULD / MAY keywords. Every requirement has a reason, acceptance scenarios use GIVEN/WHEN/THEN. Auto-validated for completeness with coverage scoring.

Layer 2 Superpowers — Agent Orchestration

14 Rules + S.U.P.E.R analysis engine. Three AI agents (小明/Hermes/Claude) orchestrate a 9-step pipeline: spec check → S.U.P.E.R analysis → PRD → review → architecture → development → self-test → code review → final report. Each step produces auditable artifacts.

Layer 3 Harness Engineering — CI/CD & Compliance

Three-layer CI/CD pipeline (Dev → Integration → System verification). Built-in cross-compilation for ARM/RISC-V/x86-64, MISRA-C static analysis gates, and one-click ASPICE compliance evidence pack generation with traceability matrix.

Installation

Choose the installation method that fits your workflow. yuleOSH supports one-line install, Docker deployment, and Kubernetes Helm charts.

Option 1: One-Click Install (curl | bash)

Clone the repository and install dependencies.

git clone https://github.com/frisky1985/yuleOSH.git
cd yuleOSH
python3 -m pip install -e .
sudo ln -sf "$(pwd)/src/cli/yuleosh.sh" /usr/local/bin/yuleosh

Or without sudo — add bin/ to $PATH:

export PATH="$PATH:$(pwd)/bin"

Option 2: Docker

Run the full yuleOSH server in a container.

docker run -p 8080:8080 \
  -v $(pwd)/.yuleosh:/app/.yuleosh \
  yuleosh/server:latest

Or using docker-compose:

docker-compose up -d

Option 3: Kubernetes (Helm)

Deploy to Kubernetes with Helm charts.

# Install with Helm
helm install yuleosh ./deploy/helm/yuleosh \
  --set persistence.enabled=true \
  --set ingress.host=yuleosh.example.com

Quick Kubernetes deployment:

kubectl apply -f deploy/k8s/quickstart.yaml

CLI Reference

The yuleosh CLI provides all the tools you need for project management, spec validation, pipeline orchestration, CI/CD, and compliance.

Command Description
init [dir]Initialize project directory structure
template init <name>Create new project from starter spec template
spec validate <file>Validate OpenSpec file (SHALL/SHOULD/MAY + GIVEN/WHEN/THEN)
spec diff <old> <new>Compare two OpenSpec files and show delta
pipeline run <spec>Run full 9-step Agent pipeline on a spec
pipeline status [name]Show pipeline session status and history
ci run <layer>Run CI layer (1=Dev | 2=Integration | 3=System)
review autoAuto-review all changed files compared to git HEAD
review task <name> [kind]Review specific task (feature|bugfix|refactor|docs|config)
evidence packGenerate ASPICE compliance evidence pack
ui startLaunch Dashboard at http://localhost:8080
stats [--json]Show project statistics and metrics
help [--examples]Show help or real usage examples

Command Examples

init — Initialize project

yuleosh init my-project
# Creates: specs/ tasks/ src/ docs/ evidence/

template init — Create from starter template

yuleosh template init my-embedded-app
# Creates full project with starter spec.md:
# docs/spec.md, src/, tests/, pyproject.toml, .gitignore

Use --from to scaffold from an existing template project:

yuleosh template init my-sensor --from templates/ble-sensor

spec validate — Validate OpenSpec requirements

yuleosh spec validate docs/spec.md
# → Coverage: 85% (SHALL: 40% · Reason: 20% · Scenarios: 25%)
#   PASS: 3 requirements, 2 scenarios validated

spec diff — Track requirement changes

yuleosh spec diff docs/spec.md docs/spec-v2.md
# → Added: Req-004 (1 SHALL), Req-005 (2 SHALL, 1 SHOULD)
#   Removed: none
#   Modified: Req-001 (SHALL count: 2 → 3)

pipeline run — Full Agent pipeline

yuleosh pipeline run docs/spec.md
# → [小明] Spec check: 3 requirements found
#   [小明] S.U.P.E.R analysis: urgency=high, impact=medium
#   [Hermes] PRD written: 3 feature sections
#   [Hermes] Review: all requirements coverable
#   [Claude] Architecture: 3 modules defined
#   [Claude] Development: src/ populated, tests/ scaffolded
#   [Claude] Self-test: 6 tests pass
#   [Hermes] Code review: 2 minor suggestions
#   [小明] Final report: session saved to .osh/sessions/

ci run — Run CI layers

yuleosh ci run 1    # Dev Verification: unit tests + coverage ≥ 40%
yuleosh ci run 2    # Integration: cross-compile + cppcheck + integration tests
yuleosh ci run 3    # System: E2E tests + version check + evidence pack

review — Code review by AI agents

yuleosh review auto                      # Auto-review all git changes
yuleosh review task add-temp-sensor feature  # Feature review: 4 agents
yuleosh review task fix-bug bugfix          # Bugfix review: 2 agents

evidence pack — Generate compliance artifacts

yuleosh evidence pack
# → traceability-matrix.md     (Req ↔ Design ↔ Code ↔ Test)
#   requirement-coverage.md    (Per-requirement coverage)
#   code-coverage-report.md    (Line/condition metrics)
#   review-log-summary.md      (Human-readable audit trail)
#   review-log.json             (Machine-readable records)
#   compliance-pack.zip         (All bundled for audit)

ui start — Launch Dashboard

yuleosh ui start
# → Dashboard at http://localhost:8080
#   Custom port: OSH_PORT=9090 yuleosh ui start

stats — Project statistics

yuleosh stats
# → Pipelines: 12 total (10 completed, 2 failed)
#   CI runs: 45 (40 passed, 5 failed)
#   Reviews: 28
#   Projects: 3

yuleosh stats --json
# → {"pipelines": 12, "ci_runs": 45, ...}

help — Get help

yuleosh help                     # Show command list
yuleosh help --examples          # Show real-world usage examples

End-to-End Workflow

# 1. Initialize from template
yuleosh template init my-sensor-project
cd my-sensor-project

# 2. Validate requirements
yuleosh spec validate docs/spec.md

# 3. Run Layer 1 CI (unit tests)
yuleosh ci run 1

# 4. Run full Agent pipeline
yuleosh pipeline run docs/spec.md

# 5. Auto-review changes
yuleosh review auto

# 6. Run full CI suite
yuleosh ci run 2
yuleosh ci run 3

# 7. Generate compliance pack
yuleosh evidence pack

# 8. Launch Dashboard
yuleosh ui start

API Reference

All API endpoints are mounted under /api/v1/ and return JSON. Every response follows the format { "ok": true, "data": {...} } or { "ok": false, "error": "..." }.

GET /api/v1/health

Comprehensive system health monitoring — DB connectivity, store status, disk space, uptime.

curl http://localhost:8080/api/v1/health
POST /api/v1/wizard/complete

Mark first-run wizard as completed.

POST /api/v1/spec/validate

Validate an OpenSpec file. Body: { "path": "docs/spec.md" }

curl -X POST http://localhost:8080/api/v1/spec/validate \
  -H 'Content-Type: application/json' \
  -d '{"path":"docs/spec.md"}'
POST /api/v1/spec/diff

Diff two OpenSpec files. Body: { "old": "...", "new": "..." }

POST /api/v1/pipeline/run

Run the 9-step Agent pipeline. Body: { "spec": "docs/spec.md", "name": "optional-name" }

GET /api/v1/pipeline/status

List all pipeline sessions.

POST /api/v1/ci/run/{layer}

Run a CI layer (1, 2, or 3).

curl -X POST http://localhost:8080/api/v1/ci/run/1
GET /api/v1/ci/runs

List all CI run results.

POST /api/v1/review/auto

Auto-review all changed files.

POST /api/v1/review/task

Review a specific task. Body: { "task": "add-sensor", "kind": "feature" }

GET /api/v1/review/list

List all review sessions.

POST /api/v1/evidence/generate

Generate compliance evidence pack.

GET /api/v1/evidence/files

List generated evidence files.

GET /api/v1/evidence/pack

Download the compliance ZIP pack.

GET /api/v1/project

List all projects.

curl http://localhost:8080/api/v1/project
POST /api/v1/project

Create a new project. Body: { "name": "my-project", "description": "..." }

GET /api/v1/project/{name}

Get a specific project by name.

GET /api/v1/project/stats

Aggregate project statistics across all store tables.

GET /api/v1/stats/overview

Aggregated counts of everything — pipelines, CI, reviews, evidence.

curl http://localhost:8080/api/v1/stats/overview
GET /api/v1/stats/trends?period=daily&days=7

Daily or weekly trends. Supports period (daily|weekly) and days params.

GET /api/v1/notify/config

Get current notification configuration (Feishu, email, webhook).

PUT /api/v1/notify/config

Update notification config. All fields optional.

POST /api/v1/apikeys

Generate a new API key. Body: { "label": "My Key" }

GET /api/v1/apikeys

List API keys (prefix + label, never the full key).

DELETE /api/v1/apikeys/{id}

Revoke an API key by ID.

POST /api/v1/webhooks/github

Receive GitHub push webhook events and trigger CI Layer 1.

Configuration

yuleOSH is configured through environment variables. The table below lists all supported variables with defaults.

Variable Default Description
OSH_HOME project root Project root directory path
OSH_PORT 8080 Dashboard HTTP server port
OSH_AUTH disabled Enable auth enabled or disabled
OSH_ADMIN_PWD admin Admin password when auth is enabled
OSH_DB_PATH .yuleosh/store.db SQLite database file path
OSH_LOG_LEVEL INFO Logging level: DEBUG, INFO, WARN, ERROR
OSH_CI_COVERAGE 40 CI Layer 1 coverage threshold (%)
OSH_SPEC_THRESHOLD 80 Spec validation coverage threshold (%)
OSH_REVIEW_THRESHOLD 80 Review coverage threshold (%)
OSH_SESSIONS_DIR .osh/sessions Pipeline session artifacts directory
OSH_EVIDENCE_DIR .osh/evidence Compliance evidence directory
OSH_CI_DIR .osh/ci CI results directory
DESIGNATED_TESTERS (empty) Comma-separated designated tester IDs

Frequently Asked Questions

What is yuleOSH and who is it for?
yuleOSH (you-unify-Lifecycle-of-OpenSpec-Superpowers-Harness) is an embedded AI development platform combining OpenSpec requirements format, AI Agent pipeline (3 agents, 9 steps), and 3-layer CI/CD. It's built for embedded MCU/SoC teams needing ASPICE compliance, AI-assisted development, and automated CI/CD.
Do I need AI API keys?
No. The Agent pipeline simulates agents with template-based artifacts — no external AI API keys required. The platform is fully functional out of the box.
Can I use yuleOSH with existing projects?
Yes. Run yuleosh init in your existing project directory to add the required structure, then write your OpenSpec in docs/spec.md. Existing files are not modified.
What are the system requirements?
Python 3.10+, Git 2.20+, macOS/Linux (Windows via WSL). Optional: pytest + coverage (CI Layer 1), cppcheck (Layer 2 static analysis).
What is OpenSpec format?
OpenSpec is a structured markdown format using RFC 2119 keywords (SHALL/SHOULD/MAY) plus GIVEN/WHEN/THEN acceptance scenarios. Each requirement must have at least one SHALL statement and a Reason section. Scenarios must have GIVEN, WHEN, and THEN clauses.
What does spec coverage score mean?
Coverage score (0–100%) measures spec completeness: SHALL statements (40%), Reason sections (20%), complete scenarios (40%). The pass threshold is 80%.
What are the 4 review agents?
Architecture (module coupling, function length), Domain Modeling (mutable defaults), Code Style (docstrings, formatting), Coverage Guardian (≥80% line coverage). Task kinds determine which reviewers run: feature=all 4, bugfix=style+coverage, refactor=architecture+style+coverage, docs=pass, config=style.
How do I run tests?
python3 -m pytest tests/ -v from the project root. Layer 1 CI runs tests with coverage gating. For available templates, see the templates/ directory with pre-built sensor, CAN bus, and MCU firmware projects.
What's in the ASPICE compliance pack?
Run yuleosh evidence pack to generate: traceability-matrix.md (Req ↔ Design ↔ Code ↔ Test), requirement-coverage.md, code-coverage-report.md, review-log-summary.md, review-log.json, and compliance-pack.zip — all bundled for ASPICE auditor submission.
